# Firestore payment-ledger security audit

- Platform: Angular 21, Firebase Auth, AngularFire modular Firestore SDK.
- Writes: owner-scoped sale balances, payment entries, customer ledger summaries.
- Authentication: all paths require `request.auth.uid` to match the path UID and `ownerId`.
- Queries: no payment-ledger reads are currently issued; writes use direct document paths.
- Default access: denied.
- Schema pollution: blocked with `keys().hasOnly(...)`.
- Negative/overflow amounts: blocked by `validMoney`.
- PII separation: customer data is nested beneath the authenticated user's document and is owner-readable only.
- Ownership hijacking: path UID and document `ownerId` must both match the authenticated UID.
- Immutable deletion: payment and ledger deletion is denied.
- Create/update validation: the same domain validator is called for both operations.
- Public list exploit: denied because every read requires owner authentication.
- Type-juggling and required-field omission: blocked by validators.
- Timestamp manipulation: server timestamp type is required; recency is not enforced because offline queued writes may resolve later.
- Remaining risk: Firebase CLI syntax validation was not run because CLI access was declined.
